# Prompt: DNS-less Local RP Identity for LinkKeys

You are working in the LinkKeys repository. Design and implement DNS-less local
RP identity so locally installed apps can use LinkKeys as their authentication
system without running a DNS-pinned RP or exposing a public RP process. Treat
security of LinkKeys users and domains as the highest priority.

This file is intentionally detailed. It should contain enough context to start a
multi-part implementation without relying on prior chat history.

## Repository Rules to Preserve

- `liblinkkeys` is a pure library: no I/O, no database, no network. Protocol
  types, crypto, serialization, claims logic, and verification helpers belong
  here when they are pure and WASM-viable.
- `linkkeys` is the server binary. CLI, database, network, HTTP, TCP, and domain
  policy persistence belong here.
- CSIL is the source of truth for protocol datatypes and service interfaces.
  Generated Rust goes in `crates/liblinkkeys/src/generated/`; do not hand-edit
  generated files.
- csilgen does not need new responsibilities for this project. It should keep
  generating types, typed clients, server traits/routers, and transport seams.
  Hand-maintained SDKs own runtime crypto, storage callbacks, browser redirect
  ergonomics, and transport wiring. The CSIL-RPC *client* for each SDK language
  is generated by csilgen; the SDK supplies only the TCP socket transport.
- Maintain SQLite and PostgreSQL compatibility for server persistence.
- Generated code and SDKs should be reproducible and locally testable.
- User-facing strings (including error pages added by this work) go through the
  i18n catalog. Never hardcode user-facing text.

## Problem

Normal RP identity is DNS-pinned: the RP has a domain, DNS proves control, and
the IDP can negotiate with that RP identity. That is too much operational weight
for local apps such as LAN jukeboxes, desktop tools, or self-hosted services
running only on loopback/LAN.

The goal is not anonymous or identity-free login. The goal is a different RP
anchor:

```text
local_rp_id = fingerprint(local_rp_signing_public_key)
```

The local app should be able to redirect a browser to a LinkKeys domain, receive
a signed/encrypted result, and verify that result locally. LinkKeys domain admins
must decide whether this local-RP mode is allowed at all, and the default should
be admin approval.

## Core Decisions

### RP Identity Is the Signing Key

The local RP identity is the fingerprint of the local RP signing public key,
similar to SSH host identity. The key is the app identity for this mode.

Why:

- DNS is intentionally unavailable as an identity anchor.
- The app still needs a durable identity that admins can approve, deny, revoke,
  and audit.
- Public key fingerprints are understandable and already familiar from SSH.
- If the key is lost or rotated, the app is a new identity. That is acceptable
  for this mode.

### Fingerprint Format: Existing Hex String

The fingerprint format is the existing LinkKeys format: SHA-256 of the raw
public key bytes, hex-encoded, exactly as `liblinkkeys::crypto::fingerprint`
produces today and as DNS `fp=` records and TLS pinning already consume.

Do not introduce a second fingerprint encoding. The earlier draft's
`bytes .size 32` is dropped. In CSIL, fingerprints are `text` (hex), matching
existing types. For human display (consent screens, admin lists), show a short
prefix (first 16 hex chars) with the full value available on demand.

### One Signing Key and One Encryption Key

A local RP has exactly one signing key and one encryption key.

- Default lifetime: 10 years.
- Rotation is a deliberate operator event.
- Rotation creates a new RP identity and requires re-approval at every LinkKeys
  domain that should allow the app.
- There is no continuity-of-key story in the first version.
- Apps that need stable domain identity or serious organization-level key
  continuity should run a normal DNS-pinned RP or IDP.

Why:

- This keeps the local mode simple and auditable.
- Automatic rotation without a DNS/domain anchor would create confusing trust
  transfer semantics.
- A stable long-lived local key matches the SSH-style mental model.

### Encryption Key Is Separate, Not Derived

The signing key remains the identity. The encryption public key is bound to that
identity by a signed local RP descriptor.

Do not algebraically derive the encryption key from the signing key.

Why:

- Ed25519 and X25519 conversion support differs across language ecosystems.
- Conversion is a footgun for SDK implementers.
- Separate signing and encryption key material keeps protocol intent explicit.
- A signed descriptor gives the same binding without relying on cross-language
  conversion behavior.

### Signatures Use the Envelope Pattern

Every new signed structure uses the envelope pattern the codebase already uses
for `SignedIdentityAssertion`: the payload is CBOR-encoded once, the signature
is computed over those exact bytes, and the envelope ships
`{ payload: bytes, signature: bytes }`. Verifiers verify the shipped bytes and
only then decode them.

Never put a `signature` field inside the structure it signs. That would force
every SDK language to reproduce byte-identical canonical CBOR (map ordering,
definite lengths) to re-serialize "the struct minus the signature", which is
the single biggest cross-language conformance footgun available to us.

### Signature Context Strings Are Mandatory

Every signature is computed over `context || payload_bytes` with a fixed
context string unique to the structure:

- `linkkeys-local-rp-descriptor`
- `linkkeys-local-rp-login-request`
- `linkkeys-local-rp-callback`
- `linkkeys-local-rp-ticket-redemption`

A signature over one structure must never verify as another. Signatures are
Ed25519, period. There is no signature versioning story and there must never be
a "v2" of any signature scheme: algorithm agility lives in cipher negotiation
(below), and any future signature-algorithm change is a new protocol mode with
new context strings, not a version bump of this one.

### Cipher Suites Are Negotiated

The AEAD suite used to encrypt the callback payload is negotiated, not fixed:

- The local RP descriptor advertises the AEAD suites the SDK supports, in
  preference order.
- The IDP picks one it supports and records the chosen suite in the callback
  header, bound into the AEAD associated data so it cannot be tampered.
- Mandatory-to-implement baseline: `aes-256-gcm` — this is the existing
  sealed-box construction in `liblinkkeys::crypto` (ephemeral-static X25519 +
  HKDF-SHA256 + AES-256-GCM with context AAD and low-order-point rejection).
  Reuse it; do not invent a parallel construction.
- Optional second suite: `chacha20-poly1305` (same KDF and binding, different
  AEAD).

Additionally, as part of this work, retrofit suite negotiation onto **all**
existing LinkKeys encrypted-payload uses, not just local RP. The current
implied-AES-256-GCM behavior becomes "negotiated, with aes-256-gcm as the
baseline". Prod is pre-alpha: do a hard cutover, no backward-compatibility shim
for the un-negotiated format. Everything gets redeployed together; third-party
deployments update on their own schedule.

Do not invent a new negotiation mechanism for the S2S paths: the CSIL already
has `AlgorithmSupport { signing, ? encryption }` exchanged via the `Handshake`
service. The retrofit means populating and honoring the `encryption` list there
and threading the chosen suite into the sealed-box call sites. The local RP
browser path has no handshake (the browser carries the payload), which is why
the descriptor advertises suites instead — same registry, two advertisement
surfaces.

### Browser-only Flow

This mode is browser-only for the first version. The SDK must return a redirect
URL. The application decides whether to HTTP redirect, display the URL, open a
browser, or embed that flow in its own web UI.

Do not call the helper `open_browser`; that name is wrong for server-side web
apps like `ichoi`. The helper should be shaped as "build/begin local login and
return the URL plus state".

Why:

- The browser is the thing carrying the user to their LinkKeys domain.
- The app might already be a browser app, a LAN web app, a desktop app, or a CLI.
- Opening a browser is a UX decision outside the LinkKeys SDK.

### The Browser Is Only a Carrier

The browser carries signed and encrypted blobs. It is not trusted with protocol
state beyond ordinary redirect mechanics.

The local RP signs the request. The IDP signs the response and encrypts it to the
local RP encryption key. The SDK verifies signatures, issuer binding, audience,
expiry, nonce, state, revocations, and claim signatures before returning a
result to app code.

Why:

- The callback URL may be LAN-only or loopback-only.
- Browser parameters are attacker-controlled unless every sensitive part is
  signed or encrypted.
- The SDK should make the secure path easier than a hand-rolled integration.

### Callbacks Carry a Claim Ticket, Not Claims

The encrypted callback payload contains the verified identity (user id, user
domain) plus a **claim-get ticket** — it does not contain the claims
themselves. The SDK redeems the ticket directly with the user's IDP over TCP
CSIL-RPC to fetch the claims.

Ticket semantics:

- Bound to the local RP fingerprint. Redemption requests are signed with the
  local RP signing key (context `linkkeys-local-rp-ticket-redemption`), so a
  stolen ticket is useless without the RP's private key.
- Valid for a bounded window, default 1 hour. Multi-use within the window, so
  the app can retry or refresh. Each redemption returns the claim *values* as
  they are at redemption time; the claim *set* is frozen to what the user
  consented to. If the user edits their email between consent and redemption,
  the app sees the new value. This is deliberate.
- Checked against the local RP's approval status at every redemption: revoking
  the RP kills its outstanding tickets.
- Redemption is TCP CSIL-RPC only. There is no HTTP redemption route. TCP is
  the future; this flow should pull integrators toward it.

Why:

- The browser round-trip stays small — the callback fits comfortably in a GET
  redirect query parameter, which removes the response-size problem entirely
  (no SAML-style form POST needed).
- Claims move over a direct, verifiable RP↔IDP channel instead of through the
  browser.
- It matches where LinkKeys is going: server-to-server traffic on TCP.

### Callback URL Rules

- `http` and `https` are both allowed. A DNS-less LAN/loopback RP cannot be
  required to have a certificate; the payload is encrypted and signed, so the
  transport back to the app does not need TLS for protocol security.
- Every other scheme is rejected (`javascript:`, `data:`, `myapp:`, custom
  mobile-app schemes — all of them). On rejection the IDP shows a friendly,
  non-technical, i18n-cataloged error to the user, something in the spirit of:
  "This app asked us to send your login somewhere that isn't a web page, so we
  stopped. Nothing happened to your account." Wording goes through the i18n
  catalog like all user-facing strings.
- Consequence, stated openly: a native mobile app cannot take a login via this
  mode (custom URL schemes are the mobile pattern, and they are rejected).
  That is fine — there is no good use case for it in this mode, and it is a
  non-goal.
- The IDP must not redirect users to the callback URL of a pending, denied, or
  revoked local RP with error details; show the error on the IDP page instead.

### Domain Admin Approval Is the Default

Each LinkKeys domain should have policy for local RP mode:

- disabled
- admin approval required, default
- optionally allow-by-default for domains that explicitly choose that risk

Admin approval keys on the local RP fingerprint alone. The signed descriptor's
metadata (app name, optional local domain/origin hint, keys, timestamps) is
display/audit context. If a known fingerprint shows up with changed metadata,
surface a drift warning to admins; do not treat metadata as part of identity.

Pending-queue protection: a pending approval entry is created only when an
**authenticated user of the domain** actually attempts a login through the
unknown local RP. Anonymous requests never create queue entries. Dedupe pending
entries by fingerprint, and cap pending entries per domain so the queue cannot
be flooded.

Why:

- A domain admin, not a random app developer, decides whether users may log in to
  local RPs.
- Local apps have no DNS proof. Admin approval is the replacement control.
- This makes local mode opt-in at the domain boundary.
- Anything that lets app-name influence approval decisions quietly promotes
  attacker-chosen display text toward identity.

### Consent UI Requirements

The consent screen for a local RP login must show:

- the app name, clearly labeled as **unverified, chosen by the app**
- a plain-language "this app is not DNS-verified" warning
- the callback host the browser will be sent back to
- the short fingerprint, with the full fingerprint available
- first-seen date for this fingerprint at this domain

App name is display/audit metadata, not identity. Users will trust whatever
name is shown; the UI has to carry the skepticism for them.

### Default Claim Set

SDK `begin_local_login` defaults, when the caller does not specify claims:

- requested claims: `display_name`, `email`, `handle`
- required claims: `handle`

These are the existing claim names in the codebase (snake_case). The point is
that a developer gets a usable "identity" out of the box with zero claim
configuration, while the SDK still accepts any other requested/required claim
setup explicitly.

## Flow

1. App developer calls `generate_local_rp_identity(config)` once at
   install/setup time.
2. App stores the returned signing key, encryption key, descriptor, and
   fingerprint wherever it stores app secrets/configuration. The SDK provides
   raw byte helpers so the developer does not invent encodings.
3. User enters/selects their LinkKeys domain in the app.
4. App calls `begin_local_login(config)` with:
   - local RP identity/key material
   - app name
   - optional local domain/origin hint
   - callback URL
   - requested claims (default `display_name`, `email`, `handle`)
   - required claims (default `handle`)
   - now
5. SDK generates nonce/state, builds and signs the login request (envelope +
   context string), and returns a redirect URL for the user's LinkKeys domain
   **plus a pending-login state object**. The app must persist that pending
   state (e.g. in its server-side session tied to the browser) and supply it to
   `complete_local_login`. It is single-use: discard it after one completion
   attempt.
6. App redirects the user's browser to that URL.
7. IDP verifies the login request: envelope signature with the included local
   RP signing public key, descriptor envelope signature, descriptor/encryption
   key binding, fingerprint equals SHA-256 hex of the signing public key,
   callback URL scheme is http/https, timestamps within bounds.
8. IDP checks domain policy for local RP mode:
   - disabled: reject
   - unknown fingerprint and admin approval required: after the user
     authenticates, create/refresh a pending entry (authenticated users only,
     deduped, capped) and show pending status
   - approved fingerprint: continue
   - revoked/denied fingerprint: reject (error shown on the IDP page, no
     redirect to the callback URL)
9. User authenticates and consents according to existing LinkKeys claim policy,
   with the consent UI requirements above.
10. IDP creates a callback payload: `{ user_id, user_domain, claim_ticket,
    fingerprint (audience), callback_url, nonce, state, issued_at, expires_at }`,
    CBOR-encoded, signed by domain keys (envelope + context string), then
    encrypted to the local RP encryption public key using the negotiated AEAD
    suite. The cleartext callback header (fingerprint, nonce, state, chosen
    suite, timestamps) is routing/decryption metadata only and is bound as AEAD
    associated data; the authoritative copies are inside the ciphertext.
11. Browser is GET-redirected to the local callback URL with the encrypted
    payload.
12. App passes the callback data, the pending-login state from step 5, the URL
    the callback actually arrived at, and `now` to `complete_local_login`.
13. SDK:
    - decrypts the payload (rejecting an unadvertised suite)
    - fetches domain public keys for **the domain the login was begun with**
      over TCP CSIL-RPC, pinned via DNS `fp=` records
    - always fetches revocations for every domain whose keys it fetches over
      TCP CSIL-RPC — this is not suppressible by any server-advertised
      "recent revocations available" hint. A `get-revocations` RPC error is
      fatal: the SDK fails closed rather than verifying against an unfiltered
      key set
    - verifies the IDP envelope signature against those keys
    - verifies issuer binding: the payload's `user_domain` equals the domain
      from the pending state, and the keys used were fetched for that domain
    - verifies audience equals the local RP fingerprint
    - verifies callback URL binding against the URL the payload arrived at
    - verifies nonce/state against the pending state
    - verifies issued/expires timestamps (with bounded clock-skew tolerance,
      default ±300 seconds; callback lifetime is short, default 5 minutes)
    - treats the verified callback payload, not the ticket-redemption
      response, as the source of truth for identity: the SDK returns
      `payload.user_id`/`payload.user_domain` (domain-signed,
      envelope-verified, issuer-bound) as the user's identity, and the
      verified `payload.user_domain` — never a value read off the redemption
      response — is the subject-domain used for every downstream per-claim
      signature check
    - redeems the claim ticket with the IDP over TCP CSIL-RPC, signing the
      redemption request with the local RP signing key
    - asserts the ticket-redemption response's `user_id`/`user_domain` equal
      the verified payload's `user_id`/`user_domain`; a mismatch is fatal —
      the redemption response corroborates the payload's identity, it never
      supplies it
    - verifies claim signatures against signer domain public keys (fetched the
      same pinned way, using the verified `payload.user_domain` as
      subject-domain) and checks revocation certificates
    - asserts every returned claim's `user_id` equals the verified payload's
      `user_id`; a mismatch is fatal
    - verifies `required_claims` (retained in the pending-login state since
      step 5, alongside nonce/state/callback URL) against the
      signature-verified claim set: every required claim name must appear
      among verified claims. An empty or insufficient claim set when
      `required_claims` is non-empty is a fatal error, not a silent partial
      success
    - returns verified user, claims, domain keys used, and metadata to app code
14. App code handles local user/session creation, deny/allow rules, caching,
    database writes, and authorization. Replay protection at the app boundary
    is the app's job: the SDK verifies equality against the pending state the
    app supplies, and the app must treat that state as single-use.

## Required Network Access

The local app does not need inbound public network access, but it needs
outbound network access to LinkKeys services:

- the user's LinkKeys domain: domain public keys, revocations, and claim-ticket
  redemption, all over TCP CSIL-RPC
- claim signer domains: public domain keys and revocations, over TCP CSIL-RPC
- DNS TXT lookups for `fp=` records of those domains
- optionally user public keys if the app wants account-key checks

Transport and trust anchor:

- SDKs use the csilgen-generated CSIL-RPC client for their language; the SDK
  hand-provides only the TCP socket transport. This is the same pattern as the
  existing csilgen transport libraries.
- Domain keys fetched this way are authenticated by pinning against DNS `fp=`
  records, the same trust anchor the existing S2S TCP path uses
  (`crates/linkkeys/src/tcp/tls.rs`). Every SDK therefore needs a DNS TXT
  lookup capability; given the SDK already requires network access, this is a
  small addition.
- The DNS resolver is configurable (e.g. a DoH endpoint for operators who want
  hardening against LAN resolver spoofing), but the default is the easiest
  path: the system resolver. Resolver spoofing is not a meaningful added risk
  for most apps in the local-LAN scenario this mode serves, and it matches
  LinkKeys' existing DNS-anchored trust model.
- Public domain-key fetch, revocation fetch, and ticket redemption must not
  require mutual TLS auth. (The TCP dispatch already allows unauthenticated
  peers for domain-key/ops calls — svc-02; ticket redemption authenticates at
  the application layer via the signed redemption request instead.)
- There are no HTTP routes for any of this. HTTPS is browser-only in LinkKeys;
  RP↔IDP traffic is TCP-first, and this mode should reinforce that, not erode
  it.

Why:

- Public keys and revocation certificates are verification inputs, not secrets,
  but they must still be *authenticated* — an unauthenticated, unpinned fetch
  would let a LAN MITM substitute domain keys and defeat every downstream
  signature check. DNS `fp=` pinning is the anchor.
- This does not allow login by itself; the payload still needs valid signatures,
  encryption, nonce/state, audience, issuer binding, and freshness, and claims
  additionally require a key-possession-proven ticket redemption.

## CSIL Work

Define as much protocol shape as reasonable in CSIL. Do not ask csilgen to
generate SDK runtime behavior; that is out of scope for csilgen. The generated
CSIL-RPC client per language is in scope and already exists as a csilgen
capability.

Likely CSIL datatypes (envelope pattern throughout; fingerprints are hex `text`
to match existing types):

```csil
;; Advertised/negotiated AEAD suites. Registry, not free text.
;; "aes-256-gcm" is mandatory-to-implement; "chacha20-poly1305" optional.
AeadSuite = text

LocalRpDescriptor = {
    app_name: text,
    ? local_domain_hint: text,
    signing_public_key: bytes .size 32,
    encryption_public_key: bytes .size 32,
    fingerprint: text,                    ;; sha256 hex of signing_public_key
    supported_suites: [* AeadSuite],      ;; preference order
    created_at: timestamp,
    expires_at: timestamp
}

SignedLocalRpDescriptor = {
    descriptor: bytes,                    ;; exact CBOR of LocalRpDescriptor
    signature: bytes                      ;; ed25519 over context || descriptor
}

LocalRpLoginRequest = {
    descriptor: SignedLocalRpDescriptor,
    callback_url: text,
    nonce: bytes,
    state: bytes,
    requested_claims: [* text],
    required_claims: [* text],
    issued_at: timestamp,
    expires_at: timestamp
}

SignedLocalRpLoginRequest = {
    request: bytes,                       ;; exact CBOR of LocalRpLoginRequest
    signature: bytes
}

;; Cleartext header: routing + decryption metadata only. Authoritative copies
;; live inside the ciphertext. The header ships as exact CBOR bytes (envelope
;; style) and those exact bytes are bound into the AEAD associated data, so no
;; SDK ever needs canonical re-serialization to verify it.
LocalRpCallbackHeader = {
    fingerprint: text,
    nonce: bytes,                         ;; protocol nonce from the request
    state: bytes,
    suite: AeadSuite,
    ephemeral_public_key: bytes .size 32,
    aead_nonce: bytes .size 12,
    issued_at: timestamp,
    expires_at: timestamp
}

LocalRpEncryptedCallback = {
    header: bytes,                        ;; exact CBOR of LocalRpCallbackHeader
    ciphertext: bytes
}

;; Plaintext inside the ciphertext is SignedLocalRpCallbackPayload — a
;; domain-signed envelope { payload: bytes, signing_key_id: text, signature }
;; around LocalRpCallbackPayload. It carries signing_key_id (like
;; SignedIdentityAssertion) because a domain holds several signing keys; the
;; RP-signed envelopes above need no key id because a local RP has exactly one.
LocalRpCallbackPayload = {
    user_id: text,
    user_domain: text,
    claim_ticket: bytes,
    audience_fingerprint: text,
    callback_url: text,
    nonce: bytes,
    state: bytes,
    issued_at: timestamp,
    expires_at: timestamp
}

LocalRpTicketRedemptionRequest = {
    claim_ticket: bytes,
    fingerprint: text,
    issued_at: timestamp
}

SignedLocalRpTicketRedemptionRequest = {
    request: bytes,
    signature: bytes                      ;; local RP signing key, its context
}

LocalRpTicketRedemptionResponse = {
    user_id: text,
    user_domain: text,
    claims: [* Claim],                    ;; existing Claim type, with its
                                          ;; existing per-claim signatures
    ticket_expires_at: timestamp
}

LocalRpVerifiedLogin = {
    user_id: text,
    user_domain: text,
    claims: [* Claim],
    domain_public_keys: [* DomainPublicKey],
    local_rp_fingerprint: text,
    issued_at: timestamp,
    expires_at: timestamp
}
```

The exact names and fields should be adjusted to fit existing LinkKeys types.
Prefer reusing existing `Claim`, domain key, revocation, timestamp, signed
envelope, and assertion types where they already exist.

Likely service/API shape:

- Public unauthenticated (TCP) domain-key fetch — already exists; confirm shape.
- Public unauthenticated (TCP) revocation fetch — already exists; confirm shape.
- Claim-ticket redemption operation (TCP only, application-layer signed).
- Local RP browser login begin/validate endpoint on the IDP side (HTTPS — this
  one is genuinely browser traffic).
- Admin APIs to list, inspect, approve, deny, and revoke local RP identities.
- Domain policy APIs/CLI/config to set local RP mode.
- Suite negotiation fields threaded through existing encrypted-payload
  operations (the protocol-wide retrofit).

## Wire Precision (Normative)

These are byte-level and placement decisions. Do not re-decide them during
implementation; if one turns out to be impossible, stop and surface it rather
than substituting a guess.

### Signature input bytes

The signature input for every new signed structure is the CBOR encoding of a
two-element array: `CBOR([ context: tstr, payload: bstr ])` — context string
first, then the exact payload bytes being enveloped. This follows the existing
house pattern of CBOR-encoding a tuple with the domain-separation tag as the
first element (see `revocation_payload` in
`crates/liblinkkeys/src/revocation.rs`, which signs
`CBOR((TAG, field, field, ...))`). Do NOT bare-concatenate `context ||
payload`; the CBOR array framing removes any ambiguity between context and
payload bytes and is trivially reproducible in every target language.

- descriptor: context `linkkeys-local-rp-descriptor`, payload = the exact CBOR
  bytes shipped in `SignedLocalRpDescriptor.descriptor`
- login request: context `linkkeys-local-rp-login-request`, payload =
  `SignedLocalRpLoginRequest.request`
- callback payload: context `linkkeys-local-rp-callback`, payload = the CBOR
  bytes of `LocalRpCallbackPayload` (which then get sealed)
- ticket redemption: context `linkkeys-local-rp-ticket-redemption`, payload =
  `SignedLocalRpTicketRedemptionRequest.request`

### Callback sealed box

The callback encryption is the existing sealed-box construction from
`crates/liblinkkeys/src/crypto.rs` (`sealed_box_encrypt`/`sealed_box_decrypt`),
extended with suite selection and header binding. Exactly:

1. Ephemeral X25519 keypair; ECDH with the local RP encryption public key;
   reject an all-zero shared secret (`reject_low_order`).
2. `context = tag || suite_id_utf8 || ephemeral_public(32) ||
   recipient_public(32)` — raw concatenation, matching `sealed_box_kdf`'s
   existing layout, with tag `linkkeys-local-rp-callback-box` and the suite id
   inserted after the tag.
3. AEAD key = HKDF-SHA256(salt = none, ikm = shared_secret).expand(info =
   context, 32 bytes).
4. AEAD = the negotiated suite (`aes-256-gcm` baseline, `chacha20-poly1305`
   optional), random 12-byte nonce.
5. AAD = `context || header_bytes`, where `header_bytes` is the exact CBOR of
   `LocalRpCallbackHeader` as shipped in `LocalRpEncryptedCallback.header`.
   (The header contains the ephemeral key and AEAD nonce; both exist before
   encryption. Only the ciphertext itself is outside the header.)

The SDK must decrypt only with a suite listed in its own descriptor and must
compare the header's authoritative twins inside the decrypted payload
(nonce/state/fingerprint/timestamps) after AAD verification.

### AEAD suite registry

Suite ids are exact strings: `aes-256-gcm`, `chacha20-poly1305`. The registry
lives in `liblinkkeys` next to `SigningAlgorithm` (same
parse/as_str/all_supported shape in `crypto.rs`). Two advertisement surfaces,
one registry: `LocalRpDescriptor.supported_suites` (browser path) and
`AlgorithmSupport.encryption` via the `Handshake` service (S2S path).

### Claim ticket bytes

A ticket is 32 random bytes, opaque. The server stores only its SHA-256 hex
(reuse `crypto::fingerprint`) plus binding columns; the raw ticket never
touches the database or logs. On the wire it is a plain `bytes` field.

### URL and parameter conventions

All CBOR-in-URL values are base64url-encoded, unpadded, exactly like the
helpers in `crates/liblinkkeys/src/encoding.rs` (`Base64UrlUnpadded`). Add
`local_rp_login_request_to_url_param`/`_from_url_param` (and callback
equivalents) beside the existing helpers.

- Begin route: `GET /auth/local-rp?signed_request=<b64url CBOR
  SignedLocalRpLoginRequest>` — mirroring the existing
  `GET /auth/authorize?signed_request=...` route shape in `web/mod.rs`.
- Callback delivery: append `encrypted_token=<b64url CBOR
  LocalRpEncryptedCallback>` to the callback URL, using the existing `?`/`&`
  separator logic (`web/mod.rs` does exactly this for the DNS-pinned flow's
  `encrypted_token` parameter — same name, same mechanics).
- Consent flows follow the existing `/auth/consent` patterns.

### Service and authorization placement

- New CSIL service `LocalRp` for wire operations:
  `redeem-claim-ticket: SignedLocalRpTicketRedemptionRequest ->
  LocalRpTicketRedemptionResponse` (TCP dispatch, unauthenticated TLS peer
  allowed — authentication is the application-layer redemption signature).
- Local RP admin operations go in the existing `Admin` service
  (`list-local-rps`, `get-local-rp`, `approve-local-rp`, `deny-local-rp`,
  `revoke-local-rp`), mapped in `required_relation_for_op`
  (`services/authorization.rs`) to `RELATION_ADMIN` with an explicit match arm
  — do not rely on the `_ =>` fallthrough.
- Domain local-RP policy follows the existing domain policy storage and
  `policy_admin_ui.rs` patterns.
- Server-side replay/nonce needs use the existing `web/nonce_store.rs`
  (DB-backed, TTL) — do not build a second store.

### SDK endpoint discovery and pinning

- Trust anchor: `_linkkeys.{domain}` TXT, format `v=lk1 fp={hex} fp={hex} ...`
  (parser precedent: `crates/liblinkkeys/src/dns.rs`).
- Endpoint discovery: `_linkkeys_apis.{domain}` TXT with `tcp=host:port`
  (`https=` exists for browser traffic; SDKs use `tcp=`).
- TLS to the TCP port: verify the server certificate's SPKI public-key
  fingerprint is in the `fp=` set, exactly as
  `crates/linkkeys/src/tcp/tls.rs` does. WebPKI validity of the cert is not
  the anchor; the pin is.
- The Rust `linkkeys-rpc-client` refuses non-public peer addresses — that is a
  server-side SSRF guard. SDKs must make address policy configurable and
  default to permissive: connecting from a LAN box to wherever
  `_linkkeys_apis` points is the entire point of this mode.

## Server Work

Implement in `linkkeys`, not `liblinkkeys`, anything involving persistence,
HTTP/TCP, CLI, database, or external network.

Likely server tasks:

- Add per-domain local RP policy:
  - disabled
  - admin approval required, default
  - optionally allow-by-default
- Add local RP approval storage:
  - fingerprint (identity key for approval)
  - signing public key
  - encryption public key
  - app name and local domain/origin hint (display/audit; drift-warn on change)
  - status: pending/approved/denied/revoked
  - created/updated/expires timestamps
  - last seen timestamp if useful
  - admin notes if existing patterns support it
- Add claim-ticket storage: ticket id/secret hash, fingerprint binding, user,
  granted claims scope, issued/expires timestamps, revocation check at
  redemption. Rate-limit redemption attempts.
- Add migrations for PostgreSQL and SQLite (per-backend directories, as usual).
- Add admin service operations and/or HTTP routes for approve/list/revoke.
- Pending-queue guard: entries only from authenticated domain users' login
  attempts, deduped by fingerprint, per-domain cap.
- Ticket redemption TCP operation: verify redemption signature against the
  approved fingerprint's signing key, check approval status (revocation kills
  tickets), return claims as of redemption time.
- Browser login route for local RP requests, including callback URL scheme
  validation (http/https only) and the friendly i18n-cataloged rejection page.
- Rate-limit the public key/revocation fetch paths; where responses are
  cacheable, say so explicitly in the protocol response shape rather than
  inventing SDK-side storage.
- Retrofit AEAD suite negotiation onto existing encrypted-payload paths
  (hard cutover, no compat shim — pre-alpha).
- Ensure service authorization maps local-RP admin operations to an appropriate
  admin relation.
- Do not log private keys, nonces, tokens, tickets, full claim values, or
  decrypted callback payloads.
- Operator docs must state plainly: revoking a local RP stops future logins and
  kills outstanding claim tickets, but sessions the app already minted are the
  app's own to manage. Revocation does not reach into app sessions.

## SDK API Shape

Every SDK should have big-config, single-function APIs first. Builder or
functional notation can exist only where it is idiomatic for that language.

Required public helpers, language-adapted:

```text
generate_local_rp_identity(config) -> LocalRpKeyMaterial

local_rp_identity_to_bytes(identity) -> bytes
local_rp_identity_from_bytes(bytes) -> LocalRpIdentity
signing_key_to_bytes(key) -> bytes
signing_key_from_bytes(bytes) -> SigningKey
encryption_key_to_bytes(key) -> bytes
encryption_key_from_bytes(bytes) -> EncryptionKey
fingerprint_to_string(fingerprint) -> text        # hex, the canonical form
fingerprint_from_string(text) -> Fingerprint

begin_local_login(config) -> (LocalLoginRedirect, PendingLogin)
complete_local_login(config) -> VerifiedLocalLogin
check_expirations(identity, now) -> ExpirationStatus
```

`begin_local_login` inputs: key material, app name, callback URL, user domain,
requested claims (default `display_name`, `email`, `handle`), required claims
(default `handle`), now. It returns the redirect URL **and** the pending-login
state the app must persist and later supply to complete. The pending state is
single-use.

`complete_local_login` inputs, spelled out because every one is load-bearing:

- key material
- the pending-login state from begin (contains nonce, state, user domain,
  callback URL)
- the raw callback data
- the URL the callback actually arrived at
- now

Transport seam: the SDK embeds the csilgen-generated CSIL-RPC client and asks
the integrating app/language runtime for only two capabilities — a TCP socket
transport and a DNS TXT lookup. Both get default implementations where the
language stdlib allows.

Avoid fluent examples as the primary docs:

```text
request.request_claim(...).request_claim(...)
```

Prefer:

```text
begin_local_login({
  key_material,
  app_name,
  callback_url,
  user_domain,
  requested_claims,   # optional, defaults above
  required_claims,    # optional, defaults above
  now
})
```

SDKs must not own application storage, sessions, database writes, or local user
authorization. They should either return verified results or call registered
callbacks with:

- user id/domain
- verified claims
- domain public keys used
- local RP fingerprint
- expirations
- enough metadata for app logs/audits without exposing secrets

## Byte Storage Helpers

SDKs must expose raw byte import/export helpers so developers can store opaque
key material in a database or file without inventing encodings.

Canonical formats:

- signing public key: 32 bytes
- signing private seed/key: 32 bytes where safe, otherwise the language's
  canonical raw private key bytes
- encryption public key: 32 bytes
- encryption private key: 32 bytes
- fingerprint: hex string, `sha256(signing_public_key_bytes)` hex-encoded —
  the existing LinkKeys fingerprint format, everywhere, with no bytes variant

Strings, JSON, PEM, DER, and base64 can exist as convenience wrappers, but they
must not be the primary storage API.

Private app keys do not directly identify users, but they control the local RP
identity. Document that apps should store them with ordinary application-secret
care.

## Expiration Helper

Each SDK needs:

```text
check_expirations(identity, now) -> ExpirationStatus
```

It should return exact datetimes and warning levels:

- notice: 180 days remaining
- warning: 90 days remaining
- critical: 30 days remaining
- expired: now >= expiry

The SDK reports facts. The app decides whether to warn admins, warn users, block
login, renew, or ignore.

generated LinkKeys CSIL code (types + CSIL-RPC client) checked into or consumed
by each package, git-usable package metadata, and local test runners.

Proposed layout:

```text
sdks/local-rp/
  conformance/
  rust/
  go/
  typescript/        # Node-only; see language matrix
  python/
  php/
  java/
  kotlin/
  csharp/
  dart/
  ruby/
  c/
  zig/
  elixir/
  swift/
  ocaml/
```

`conformance/` is a first-class deliverable, not a test afterthought: the Rust
`liblinkkeys` implementation generates JSON vectors — keys, descriptor bytes,
envelope signatures, callback headers and ciphertexts per suite, ticket
redemption requests, DNS-pin fixtures, plus tamper/negative cases — and every

toolchains where useful. Swift is least compelling and may remain untested until
a real Swift toolchain is available.

Do not add Reactorcide jobs for these SDKs yet. csilgen transport libraries do
not have those jobs yet, and local parity is enough for the first pass.

## Language Crypto Matrix

Research date: 2026-07-12. Suite baseline is aes-256-gcm (with optional
chacha20-poly1305), which is why libsodium-centric routes are avoided:
libsodium's AES-256-GCM is AES-NI-hardware-gated and not portable.

| Language | Likely SDK status | Crypto source | Notes |
|---|---:|---|---|
| Rust | Strong | `ed25519-dalek`, `x25519-dalek`, `aes-gcm`, `hkdf`, `sha2` | Already liblinkkeys' dependencies; the SDK is largely a wrapper over liblinkkeys itself. |
| Go | Strong, zero deps | stdlib `crypto/ed25519`, `crypto/ecdh` X25519, AES-GCM, `crypto/hkdf` | `crypto/hkdf` is stdlib since Go 1.24, so with aes-256-gcm there are no external deps at all. `x/crypto/chacha20poly1305` only if the optional suite is implemented. |
| TypeScript / Node | Strong, Node-only | Node `crypto` | Node stdlib covers Ed25519, X25519, AES-GCM, HKDF, randomness. **The browser is not a target.** Browser-based RP identities are an incredibly stupid idea and you should never do them: it would mean the RP private key lives in browser storage. Anyone who insists can compile liblinkkeys to WASM in a real language and own the consequences. |
| Python | Strong with dependency | `cryptography` | Python stdlib lacks Ed25519/X25519/AEAD. `cryptography` supports Ed25519, X25519, AEAD, HKDF, and raw bytes. DNS TXT needs `dnspython` or a small resolver shim. |
| PHP | Strong, stdlib only | `ext/sodium` (keys) + `ext/openssl` (AES-GCM) + `hash_hkdf()` | Bundled since PHP 7.2: sodium does Ed25519/X25519, openssl does AES-256-GCM, `hash_hkdf` is core. Top-tier fit for the self-hosted LAN web app audience (the archetypal local RP). csilgen has a PHP target with `php-typesonly`/`php-client`/`php-server` sub-targets; `php-client` gives the generated CSIL-RPC client. |
| Java | Strong on JDK 17+ | JCA/JCE standard names | Ed25519, XDH/X25519, AES-GCM, SHA-256 all in JCA on 17. **HKDF is the gap**: the standard KDF API only lands in JDK 24, so on 17 hand-roll HKDF over `HmacSHA256` (~15 lines, cover with conformance vectors). Known pain point: raw Ed25519 public key bytes ↔ `EdECPoint` (little-endian y + x-parity bit) — ship a documented helper. |
| Kotlin/JVM | Strong on JDK 17+ | Same as Java | Wrap JCA/JCE idiomatically. Android provider availability needs separate verification. |
| C#/.NET | Good with dependency | NSec for Ed25519/X25519; stdlib `AesGcm` + `HKDF` | Stdlib `ECDiffieHellman` is NIST-curves-only, so X25519 must come from NSec too. Prefer stdlib `AesGcm` over NSec's (NSec's AES-GCM is hardware-gated). |
| Dart | **Blocked for production TCP** | `cryptography_plus` (health-checked fork of dint-dev's package) | Crypto and full conformance pass, but `dart:io`'s BoringSSL-backed TLS client refuses handshakes with Ed25519 server certificates (`NO_COMMON_SIGNATURE_ALGORITHMS`) — and LinkKeys domain TLS certs are Ed25519. A Dart local RP therefore cannot reach a real IDP over the pinned TCP path until Dart's TLS stack supports Ed25519 certs or LinkKeys servers offer an alternate-key certificate. SDK is implemented and vector-conformant; the limitation is documented in its README. |
| Ruby | Strong, stdlib | bundled `openssl` gem | `OpenSSL::PKey.new_raw_private_key`/`new_raw_public_key` handle Ed25519/X25519 raw bytes; `OpenSSL::Cipher` does AES-256-GCM. No rbnacl — libsodium can't do portable AES-GCM anyway. |
| C | Strong with system lib | OpenSSL libcrypto (or mbedTLS) | Not libsodium, for the AES-GCM portability reason above. OpenSSL covers Ed25519, X25519, AES-GCM, HKDF. Build/packaging needs care as always in C. |
| Zig | Strong crypto, TLS pinning gap | `std.crypto` | Ed25519, X25519, AES-256-GCM, HKDF, SHA-256 all stdlib and vector-conformant. But `std.crypto.tls.Client` (0.14.1) exposes no peer-certificate hook, so SPKI pinning is impossible with stdlib TLS; the SDK fails closed (`error.PinnedTlsUnavailable`) rather than connecting unpinned, and the integrating app must supply a pinning-capable SecureDial until Zig's TLS client exposes the peer cert. Pin-extraction logic itself is implemented and tested. Also pin to the catalyst-tools Zig — std.crypto API churn is real. |
| Elixir | Strong, stdlib | Erlang/OTP `:crypto` | No external dependency needed: `:eddsa` (Ed25519), `:ecdh` with x25519, `:crypto_one_time_aead` (aes_256_gcm and chacha20_poly1305). HKDF is ~10 lines of HMAC. Drop the enacl/libsodium route entirely. |
| OCaml | Good with dependency | `mirage-crypto-ec`, `mirage-crypto`, `hkdf` | Resolved: mirage-crypto-ec provides Ed25519 + X25519, mirage-crypto provides AES-GCM, the `hkdf` package covers KDF — all maintained in the MirageOS ecosystem. No vendored-C route needed. Toolchain via catalyst-tools opam. |
| Swift | Deferred — no toolchain | CryptoKit | CryptoKit has the needed concepts, but no Swift toolchain exists on the dev box (probed 2026-07-13) and csilgen transport notes already mark Swift untested for the same reason. Deliberately unimplemented until a real toolchain is available; the conformance vectors make a future implementation a well-bounded task. |

## Implementation Phases

Each phase is sized for one working session and ends with all tests passing.
Do not start a phase until the previous one is complete. Within a phase, do
not touch work that belongs to a later phase. If a phase reveals a defect in
an earlier phase's output, fix it there first. Read the Wire Precision section
before phases 1–8; it is normative.

**Phase 1 — CSIL types and services.** Add the local RP datatypes (descriptor,
login request, callback header/payload, ticket redemption, suite fields),
the `LocalRp` service, the `Admin` service additions, and domain local-RP
policy shape. Regenerate:
`csilgen generate --input csil/linkkeys.csil --target rust --output crates/liblinkkeys/src/generated/`.
Done when: generated code compiles, is rustfmt-clean, and existing tests pass.
Don't touch: server logic, crypto helpers.

**Phase 2 — liblinkkeys pure helpers.** Envelope sign/verify with the context
strings, suite registry beside `SigningAlgorithm`, the callback sealed-box
variant per Wire Precision, fingerprint/nonce/state/expiry/audience helpers,
URL param helpers in `encoding.rs`, typed errors. Unit tests for every
tamper/negative case in the test list that is pure.
Done when: `cargo test -p liblinkkeys` passes; no I/O added to the crate.

**Phase 3 — suite negotiation retrofit.** Populate and honor
`AlgorithmSupport.encryption` in the `Handshake` service; thread the chosen
suite through the existing sealed-box call sites. Hard cutover, no compat
shim. Enumerate the call sites first (grep for `sealed_box_encrypt` /
`EncryptedToken`) and list them in the session notes before changing any.
Done when: both-backend test suites pass; S2S e2e harness still logs in.

**Phase 4 — server persistence.** Migrations for PostgreSQL and SQLite
(per-backend directories): domain local-RP policy, approval storage, ticket
storage. Diesel models, DataUtils factories, pending-queue guard logic
(authenticated-attempt-gated, deduped, capped).
Done when: `TEST_DATABASE_BACKEND=sqlite` and `=postgres` suites both pass.

**Phase 5 — TCP operations.** Confirm the unauthenticated domain-key and
revocation fetch shapes; add `redeem-claim-ticket` dispatch with redemption
signature verification, approval-status check, rate limiting. Redemption must
also reject when the ticket's user is deactivated or purged (Phase 4 finding:
user purge minimizes the row rather than deleting it, so ticket FK cascades
never fire — the status check at redemption is the backstop, and Phase 7's
admin flows should delete a purged user's outstanding tickets explicitly).
Done when: TCP dispatch tests cover the full ticket lifecycle including
revoked-RP and wrong-key rejection.

**Phase 6 — browser routes and UI.** `GET /auth/local-rp` begin route,
request/descriptor/policy validation, consent UI additions (unverified-name
warning, callback host, short fingerprint, first-seen), callback redirect with
`encrypted_token=`, scheme validation with the friendly i18n-cataloged error
page, pending/denied pages. All new user-facing strings go through the i18n
catalog.
Done when: Rocket local-client e2e covers approved/pending/denied/bad-scheme
paths.

**Phase 7 — admin surface.** `Admin` service ops + CLI affordances to
list/inspect/approve/deny/revoke with metadata drift warnings, wired to
`RELATION_ADMIN` via an explicit `required_relation_for_op` arm. Proactive
pending-entry notification is deferred to a future events system; do not build
it.
Done when: admin ops enforce the relation and CLI round-trips
approve → login-succeeds, revoke → login-and-tickets-fail.

**Phase 8 — conformance vectors.** Generate JSON vectors from the Rust
implementation into `sdks/local-rp/conformance/`: keys, descriptor bytes,
envelope signatures, callback headers and ciphertexts per suite, ticket
redemption requests, DNS TXT parsing fixtures, URL param encodings, and
tamper/negative cases for each. Write the vector schema README.
Done when: a fresh liblinkkeys test run consumes the checked-in vectors and
passes — the Rust impl is consumer zero.

**Phases 9+ — SDKs, one per session**, in priority order: Rust, Go,
TypeScript/Node, Python, PHP, Java, Kotlin, C#/.NET, Dart, C, Ruby, Zig,
Elixir, then Swift/OCaml after toolchain verification. Each SDK session:
generate types + CSIL-RPC client via csilgen for that language, implement the
crypto adapter per the language matrix row, the TCP transport + DNS TXT
lookup seams, the public API surface from "SDK API Shape", and a `tools.sh`
test subcommand.
Done when: the SDK passes the full conformance vector suite, including
negative cases, under its `./tools.sh test-local-rp-<lang>` command.

**Final phase — docs.** Operator setup, app developer setup, domain admin
policy, key expiration/rotation, revocation semantics (tickets die, app
sessions are the app's), and security tradeoffs.

## Tests and Verification

Minimum tests:

- fingerprint calculation matches existing `crypto::fingerprint` (hex) exactly
- descriptor envelope signature verifies and fails on tampering
- signature with the wrong context string fails against every other structure
- encryption public key is bound by descriptor signature
- login request envelope signature verifies and fails on tampering
- callback decrypts only with the local RP encryption key
- callback header tampering (fingerprint/nonce/state/suite swap) fails AAD
- suite negotiation: IDP never selects an unadvertised suite; SDK rejects an
  unadvertised suite in the header
- nonce/state replay and mismatch fail
- issuer binding: payload from a different domain than begun-with fails
- callback URL binding: payload delivered to a different URL fails
- expired descriptor/request/callback fails; skew tolerance boundaries exact
- `check_expirations` returns exact threshold states
- ticket redemption: possession proof required, wrong key fails, expired ticket
  fails, revoked local RP fails, multi-use within window works
- unknown local RP under default policy is pending/admin-approval, not allowed
- pending entries: anonymous attempts create nothing; deduped; per-domain cap
- denied/revoked local RP is rejected, with no redirect to its callback URL
- non-http(s) callback URL rejected with the friendly error page
- approved local RP succeeds
- public domain-key and revocation fetch work without mutual auth over TCP
- DNS `fp=` pin mismatch fails key acceptance
- SDK byte/hex round trips across conformance vectors
- every SDK passes the shared conformance vectors, including negative cases

Run local tests like existing linkkeys/csilgen workflows. Do not add Reactorcide
jobs for the new SDKs in the first pass.

## Security Review Checklist

- No DNS-less RP is trusted merely because it says an app name.
- App name is display/audit metadata, not identity; approval keys on
  fingerprint only, with metadata drift warnings.
- Fingerprint is the existing hex SHA-256 of canonical signing public key bytes.
- All signatures use the envelope pattern over exact shipped bytes.
- All signatures use mandatory per-structure context strings; Ed25519 only; no
  signature versioning, ever.
- Fetched domain keys are authenticated via DNS `fp=` pinning over TCP
  CSIL-RPC — the SDK never trusts unpinned key material.
- Issuer binding: the response domain equals the domain the login began with.
- Returned identity comes from the signed, envelope-verified, issuer-bound
  callback payload (`payload.user_id`/`payload.user_domain`), never from the
  ticket-redemption response. The SDK asserts the redemption response's
  `user_id`/`user_domain` equal the payload's (fatal on mismatch) and uses the
  verified `payload.user_domain` — never the redemption response's — as the
  subject-domain for per-claim signature verification.
- Every returned claim's `user_id` equals the verified payload's `user_id`;
  a mismatch is fatal.
- `required_claims` is enforced client-side, not merely trusted from the
  server: the SDK retains `required_claims` in the pending-login state and,
  at completion, verifies every required claim is present among the
  signature-verified claims. An empty or insufficient claim set when
  `required_claims` is non-empty is a fatal error.
- Signing private key and encryption private key are never logged.
- Nonce/state are random, verified, and single-use at the app boundary.
- Timestamps are bounded, verified, and skew-tolerant within a fixed window.
- Callback audience is the local RP fingerprint.
- Callback URL is scheme-validated (http/https only) and bound into the signed
  payload; the SDK checks it against the actual arrival URL.
- Cleartext callback header is bound as AEAD associated data.
- AEAD suite is negotiated from the descriptor's advertised list only.
- Claim tickets are fingerprint-bound, possession-proven at redemption,
  time-boxed, rate-limited, and dead on RP revocation. Redemption verifies the
  ticket's issuance fingerprint equals the possession-proven caller
  fingerprint: a valid signature from a *different* approved RP must not
  redeem another RP's ticket.
- Claim signatures verify against signer domain public keys.
- Revocation certificates are checked. The SDK always fetches
  `get-revocations` for every domain whose keys it fetches, regardless of any
  server-advertised freshness/availability hint, and treats a
  `get-revocations` RPC error as fatal — it fails closed rather than
  proceeding with an unfiltered key set.
- Public key fetch without mutual auth exposes only public verification
  material.
- Admin approval is the default domain policy; pending queue is
  authenticated-attempt-gated, deduped, and capped.
- Rotation creates a new identity; do not imply continuity.
- SDKs return verified data to app code but do not create local sessions or
  write app databases.

## Non-goals for the First Pass

- Do not implement cross-domain DNS negotiation for local RPs.
- Do not implement automatic key continuity or transparent rotation.
- Do not make csilgen generate runtime SDKs (generated CSIL-RPC clients are
  fine; that capability already exists).
- Do not make SDKs own app storage/session/database behavior.
- Do not support custom URL schemes or native mobile-app callback flows; a
  local mobile app cannot take a login via this mode, by design.
- Do not support browser-resident RP identities in the TypeScript SDK or
  anywhere else.
- Do not add HTTP routes for key fetch, revocation fetch, or ticket
  redemption; those are TCP CSIL-RPC only.
- Do not add Reactorcide SDK jobs before csilgen transport libraries have an
  equivalent pattern.
- Do not require inbound public network access to the local app.

## Open Items to Resolve During Implementation

- Exact CSIL field names and reuse of existing assertion/domain-key/envelope
  types (the sketch and Wire Precision constrain semantics; renaming to match
  existing conventions is fine, inventing different semantics is not).
- Admin UX layout for the pending local RP approval list (list/inspect/act).
  Proactive notification is decided: deferred to a future events/avenues
  system; not in this pass.
- How much caching the SDK should expose for domain public keys and revocations
  without owning app storage (protocol-level cache hints preferred).
- Ed25519 verification strictness across SDK backends (dalek `verify` vs
  `verify_strict`, OpenSSL, libsodium, JCA, BoringSSL, OTP): implementations
  genuinely differ on malleability edge cases (non-canonical S, small-order A)
  that the conformance vectors do not currently cover. All signatures in this
  protocol are produced by honest signers over envelope bytes, so acceptance
  divergence has no known exploit here, but adding malleability-edge vectors
  would pin the behavior deliberately rather than by backend accident.

## Decided (previously open)

- Fingerprint format: the existing hex-encoded SHA-256 string, unchanged.
- DNS resolver: configurable in SDKs, defaulting to the system resolver — the
  easiest path. LAN resolver spoofing is an accepted, documented tradeoff for
  this mode.
- Ticket redemption returns claim values current at redemption time; the claim
  set is frozen at consent.
- Suite-negotiation retrofit ships as a hard cutover; all owned deployments are
  simply redeployed together. This is alpha software; uncoordinated third-party
  deployments update on their own schedule.

## Post-implementation security review (2026-07-14)

A security review of the SDK verification chain and the server-side
ticket-redemption path found that identity and authorization were
under-bound in five places. All five are now normative requirements above
(Flow step 13 and the Security Review Checklist); this subsection exists only
so the history of why they were added stays legible.

1. **Identity source.** The SDK's returned identity is
   `payload.user_id`/`payload.user_domain` from the signed, envelope-verified,
   issuer-bound callback payload — never the ticket-redemption response. The
   SDK asserts the redemption response's `user_id`/`user_domain` match the
   payload's (fatal on mismatch), and uses the verified `payload.user_domain`,
   never the redemption's, as the subject-domain for per-claim signature
   verification. Previously the redemption response could disagree with the
   payload and the mismatch went unchecked.
2. **Claim ownership.** Every returned claim's `user_id` is asserted equal to
   the verified payload's `user_id` (fatal on mismatch), closing a path where
   a claim for one user could be attributed to another.
3. **`required_claims` enforcement.** The SDK now retains `required_claims` in
   the pending-login state from `begin_local_login` and, at completion,
   verifies every required claim is present among signature-verified claims.
   An empty or insufficient claim set when `required_claims` is non-empty is
   fatal. Previously `required_claims` was advisory only.
4. **Revocation fetch is unconditional.** The SDK always fetches
   `get-revocations` for every domain whose keys it uses, regardless of any
   server-advertised "recent revocations available" flag, and treats a
   `get-revocations` RPC error as fatal (fail closed). Previously a
   server-controlled flag could suppress the fetch entirely.
5. **Ticket redemption is RP-bound at the server.** Redemption now verifies
   the ticket's issuance fingerprint against the possession-proven caller
   fingerprint, so a valid signature from a different *approved* RP cannot
   redeem another RP's ticket. Previously a valid signature from any approved
   RP was sufficient, regardless of which RP the ticket was issued to.
